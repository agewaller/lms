# 10. フェーズ設計

最初から完成形を狙わず、**段階的にリスクを取りながら** 機能を育てる。

---

## Phase 1 : 通知型（観察のみ）

### 目的
- **発注しない**。シグナルを作って Slack に投げるだけ。
- Excel/VBA/RSS の動作・信頼性を毎日観察して、地雷をすべて洗い出す。

### 実装範囲
- 銘柄監視（10 銘柄）
- `RSS_raw` への RSS 関数設置
- `modRssData` のデータアクセス層
- `modSignal.ScanAll` + 戦略 1 つ（ブレイクアウト）
- `modNotify.NotifyInfo` → Python `notify_slack.py`
- `監視ダッシュボード` の簡易表示
- ハートビート監視 + 停止通知

### 実装しないもの
- 承認 UI
- 発注（`modOrder.SendOrder` は未実装 or stub）
- 建玉管理（RSS からの読みのみ、計算しない）
- 損切・利確

### 成功条件
- 営業日 5 日連続で **ブックが落ちない**
- Slack 通知の誤検知が 1 日 3 件以下
- `エラーログ` の FATAL が 0 件
- RSS 凍結を **少なくとも 1 回検知できた** ことがある（演習含む）

### 次フェーズへ進む条件
- 成功条件クリア + 通知の **意味** を自分で 100% 説明できる

---

## Phase 2 : 承認付き半自動

### 目的
- 人が押せば発注できる。ただし停止条件はすべて効いている。
- **MVP のゴール**。

### 実装範囲
- `modOrder.SendOrder` LIVE 実装（RSS 発注関数）
- 承認ボタン、取消ボタン、緊急停止ボタン
- `modRisk.PreCheckOrder` の 15 項目すべて
- `modPosition.SyncFromRss` と建玉整合チェック
- 注文管理/建玉管理/約定履歴/売買ログ/エラーログ
- 二重発注防止（`AcquireOrderLock`, `cfgMinTickInterval`）
- KillSwitch
- `cfgTradingMode` による PAPER/LIVE 切替
- Python `log_to_csv.py` による日次ダンプ
- Python `report_daily.py` による日次レポート

### 実装しないもの
- 自動発注（承認なし）
- 複数戦略同時稼働
- 外部分析エンジン連携
- バックテスト

### 成功条件
- PAPER モードで `08_test_plan.md` の T01〜T15 がすべて合格
- LIVE モードで **最小単元 1 単位** の実弾発注が 10 回連続エラーなし
- 1 ヶ月（約 20 営業日）の連続運用で FATAL 0 件
- 1 日損失上限が **1 回以上きちんと発動した** ことがある（自然 or 演習）
- 建玉不整合が 0 件

### 次フェーズへ進む条件
- 成功条件クリア + 自分がこのシステムに **慣れた** と感じる
- 承認ボタンを押す時に **迷わない** ようになる（=戦略の検証が済む）

> **Phase 2 を 3 ヶ月かけても全く問題ない**。焦らない。

## Phase 3 : 小額自動

### 目的
- 事前承認なしで発注するが、**1回・1日の損失上限が極めて厳しい**。
- 「承認ボタンを押さなくて良い範囲」を明確に定義する。

### 実装範囲
- `cfgApprovalRequired=FALSE` を **銘柄タグ単位** で許可（例：`auto_ok` タグのみ）
- 自動化される銘柄は **極小額**（例：1銘柄 5 万円以下）
- 自動化される戦略は **検証済の 1 つだけ**
- `cfgMaxLossPerDay` を Phase2 の 1/2 に引き締め
- 決済は自動、新規も自動（ただし時間帯厳格化）
- 急騰急落検知の強化（1 分で ±X% で全停止）
- 連続エラー検知の強化

### 実装しないもの
- 全銘柄の自動化
- 複雑な戦略組合せ

### 成功条件
- 自動発注経由で **1 ヶ月** 連続で致命事故 0
- 最大ドローダウンが `cfgMaxLossPerDay` × 日数の想定内
- 停止条件の発動ログがすべて意図通り

### 次フェーズへ進む条件
- 成功条件クリア + 自分のメンタルが壊れない + 家計への影響が問題ない

---

## Phase 4 : 拡張

### 目的
- 分析エンジン分離と戦略の複数化。

### 実装範囲
- Python 分析エンジンとの **明示的 I/F**（CSV or SQLite）
- 戦略の複数同時稼働（`clsStrategyXxx` を複数登録）
- 戦略ごとの損益計算
- バックテスト連動（Python で過去検証→シグナル候補→Excel 取込）
- 週次/月次レポート
- 決算カレンダー取込自動化
- 停止条件の自動学習（=やりすぎないこと）

### 実装しないもの（引き続き）
- **執行経路を VBA 以外に出す**
- **外部 API への発注**
- **他人への自動配信**

### 成功条件
- 複数戦略が相互干渉せずに動く
- 分析エンジンを停止しても執行エンジンは動く（逆は不可）
- 戦略 A/B テストができる

### 次フェーズへ進む条件
- 特になし（以降は運用改善の継続）

---

## フェーズ共通ルール

1. **フェーズは戻れる**。Phase 3 で不具合が出たら躊躇なく Phase 2 に戻す。
2. **フェーズ遷移は "終わった日の引け後" に行う**。ザラ場中に切替えない。
3. **フェーズごとに Git ブランチ or タグ** を切る。いつでも戻れるように。
4. **フェーズを上げる時に新しいパラメータを追加するのは避ける**。既存パラメータを引き締めるだけ。
5. **新しい戦略の追加はフェーズ遷移と同時にしない**。変更は 1 軸だけ。
