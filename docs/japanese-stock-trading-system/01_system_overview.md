# 01. システム全体像

## 1. システム目的

- 楽天証券 国内株（現物）の **半自動売買** を、誤発注/過剰損失を起こさずに実運用する。
- 「銘柄監視 → シグナル → 人による承認 → 発注 → 約定 → ログ → 通知 → リスク停止」
  までの **一連の業務フロー** を Excel/VBA 上で完結させる。
- 将来は分析エンジン（Python）と執行エンジン（Excel/VBA）を **分離して接続** できる構造にしておく。

## 2. 想定利用者

| 区分 | 想定 |
|---|---|
| 主ユーザー | 個人投資家本人（自分用） |
| スキル | Excel/VBA を読み書きできる中級〜上級 |
| 同時利用者 | 1名（Excel ブックは単一インスタンス前提） |
| 端末 | Windows 10/11 のデスクトップ。スリープ無効化必須。 |

## 3. 全体フロー

```
[起動] → [設定読込] → [接続確認] → [銘柄一覧読込]
   ↓
[監視ループ：N秒間隔]
   ├─ 1) RSS から株価/板/余力/建玉 取得
   ├─ 2) シグナル判定（売買禁止条件チェック含む）
   ├─ 3) リスク判定（停止条件 / 損失上限 / 同時保有数）
   ├─ 4) 候補テーブル更新（"承認待ち" 行を追加）
   ├─ 5) ダッシュボード更新
   ├─ 6) 通知送信（必要なもののみ）
   └─ 7) 例外検出（接続断 / RSS停止 / 値異常）→ 緊急停止
   ↓
[ユーザー操作]
   ├─ 承認ボタン押下 → 発注前チェック → 注文送信 → 注文ID記録
   ├─ 取消ボタン押下 → 注文取消
   └─ 緊急停止ボタン押下 → 全ルーチン停止
   ↓
[約定検知]（ポーリング or イベント）
   ├─ 注文管理シート更新
   ├─ 建玉管理シート更新
   ├─ 約定履歴 / 売買ログ書込
   └─ 通知送信
   ↓
[引け] → [日次集計] → [日次レポート出力] → [シャットダウン]
```

## 4. 主要モジュール

| モジュール | 役割 | 実装言語 |
|---|---|---|
| **Excel ブック** | データ表示・状態保持・人の操作 UI | Excel |
| **RSS データ層** | RSS関数による相場/口座データ取得 | Excel関数 |
| **VBA 中核** | シグナル判定・発注・リスク制御・ログ | VBA |
| **VBA UI** | フォーム / リボン / 承認ボタン | VBA |
| **Python 補助** | Slack通知・CSV/SQLiteログ・日次レポート | Python |
| **外部分析（将来）** | スクリーニング・予測モデル | Python |

## 5. MVP範囲（Phase 1〜2）

### Phase 1（通知型）
- 銘柄監視（10〜30銘柄）
- シグナル発生時に Slack 通知
- 発注は人が手動

### Phase 2（承認型 半自動）
- 銘柄監視 + シグナル + 承認 UI
- 承認ボタン押下で VBA から発注
- 約定確認・建玉更新・ログ
- リスク停止条件すべて実装

> **MVP のゴール**：自分が席を外していない時間帯に限り、承認ボタンだけで発注し、
> 一日の損失上限・同時保有数を確実に守って終わる、を「1ヶ月安定運用」する。

## 6. 将来拡張範囲（Phase 3〜4）

- Phase 3：小額自動（事前承認不要、ただし停止条件強化）
- Phase 4：分析エンジン連携 / 戦略複数化 / 自動レポート / バックテスト連動

## 7. 設計原則（守るべき不変条件）

1. **発注は最後の最後**。値の取得 → 判定 → リスクチェック → 承認 → 発注、の順序を崩さない。
2. **二重発注は絶対に許さない**。未約定/承認待ち/送信中フラグで多重ロックする。
3. **停止は即時、再開は手動**。停止フラグが立ったら自動再開しない。
4. **すべての発注に「理由」がある**。シグナル ID と紐づけてログに残す。
5. **データソース層と意思決定層を分離**。RSS 取得シートと判定ロジックを混ぜない。
6. **将来 Python に分離できる粒度** で関数を切る（純粋関数化を意識）。
7. **設定はコードに書かない**。`設定` シートと名前付きセルから読む。
8. **ログは "後から見れば事故が再現できる" 粒度**で残す（タイムスタンプ・値・理由）。
