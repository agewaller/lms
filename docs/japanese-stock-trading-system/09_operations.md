## 2. 立会中（09:00〜15:00）

| タスク | 頻度 | 内容 |
|---|---|---|
| ダッシュボード確認 | 常時（視野） | RUNNING / 損益 / エラー件数 |
| シグナル確認 | 発生時 | 内容が妥当か、戦略タグが合っているか |
| 承認 | 発生時 | 2 秒以上考えてから押す |
| エラー確認 | Slack 受信時 | 深刻度により即対応 |
| 停止条件監視 | 常時 | 自動停止したら原因特定 |
| 昼休み中の挙動 | 11:30-12:30 | `TimeFilter` により発注抑止（目視確認） |
| 引け前 5 分 | 14:55-15:00 | 新規抑止、決済のみ |

### 立会中の禁止事項

- **証券会社のブラウザ / MS2 本体から手動発注しない**（建玉不整合の原因）。
- **`trading.xlsm` を別名保存・閉じる・コピーしない**。
- **リスクパラメータを立会中に変更しない**（やむを得ず変更する場合は一時停止してから）。
- **別の Excel マクロを同時起動しない**（OnTime 競合）。

## 6. 障害時フロー

### 6-1. 初動（0〜1分）

1. 落ち着く。
2. **`cfgKillSwitch=TRUE`** を押す（最悪でもこれだけやる）。
3. 監視ダッシュボードの `stateHaltReason` を確認。
4. Slack 通知の内容を確認。

### 6-2. 売買停止（0〜5分）

1. `建玉管理` と MS2 本体の保有銘柄を **数値比較**。
2. 未約定注文を MS2 本体で確認。
3. 差分があれば **MS2 本体から手動で** 未約定取消。
4. 差分の再発確認を `trading.xlsm` 側でも。

### 6-3. 原因特定（5〜30分）

1. `エラーログ` シートの最終 10 行を確認。
2. `売買ログ` で時系列を追う。
3. RSS ハートビートのグラフ（`_meta` シートに記録）を確認。
4. Python `python_errors.log` を確認。
5. MS2 本体の接続・認証状態を確認。

### 6-4. 復旧条件

以下 **すべて** 満たした時のみ再開可：

- [ ] 原因が特定できている
- [ ] 建玉の整合性が取れている
- [ ] 再発防止の手当て（パラメータ調整 or コード修正）が済んでいる
- [ ] 再発しないことを PAPER モードで確認
- [ ] `エラーログ` を人が Read 済扱いにクリア（別シートに退避）

### 6-5. 再開手順

1. `cfgKillSwitch=FALSE`
2. `stateHaltReason=""`
3. `cfgTradingEnabled=TRUE`
4. Slack に **手動で** "復旧" を通知
5. 最初の数分は注視

> **復旧を急がない**。不安がある日はその日は再開しないでよい。

## 7. セキュリティ運用

| 項目 | 運用 |
|---|---|
| `trading.xlsm` | 個人ユーザーのみ読取・書込、他ユーザー不可（NTFS） |
| `secrets.ini` | 同上。Git 管理外 |
| MS2 本体のログイン | 自動ログインさせない |
| スクショ | Slack に本物の注文 ID を貼らない（マスク） |
| リモート接続 | 原則禁止。必要時は公式 VPN 経由 |
| USB 持ち出し | ログの暗号化必須 |
